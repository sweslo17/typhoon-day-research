# 本週會放颱風假嗎？做一個透明但不裝神的模型

每個颱風來之前，台灣都會出現一個臨時預測市場。有人看氣象署路徑，有人看颱風論壇，有人看隔壁縣市，有人看縣市長過去被罵的紀錄。最後大家把這些訊號混在一起，得出一句話：「我覺得明天會放。」

這篇要做的不是取代這些直覺，而是把直覺整理成一個透明模型。透明的意思是：每個輸入都看得到，每個加分都說得出來，最後輸出的是機率和原因分解，而不是神諭。

## 模型先從法規開始

颱風假不是純粹機器學習問題。法規已經告訴我們哪些訊號重要。《天然災害停止上班及上課作業辦法》第 4 條提到三種核心條件：暴風半徑四小時內可能經過，平均風力七級以上或陣風十級以上；雨量達參考基準且已致災或有致災之虞；以及地形、交通、水電供應等安全風險。

因此 baseline 模型先放入這些特徵：

- `storm_radius_within_4h`：暴風半徑是否可能四小時內經過。
- `mean_wind_mps`：平均風速。
- `max_gust_mps`：最大陣風。
- `rainfall_24h_mm`：24 小時雨量。
- `county`：縣市，未來可接縣市歷史傾向。
- `target_date`：目標日期，用來產生選舉時序特徵。

目前模型還沒有把地形、道路、土石流警戒完整接入，所以它不是正式決策模型。它是一個可解釋的 baseline。

## 為什麼要放選舉時序

選舉時序不是因為我們要假設官員一定政治操作，而是因為災害決策本來就是高可見度行政行為。越接近選舉，錯放或漏放都可能變成新聞事件。社會科學研究也長期討論災害與選舉問責的關係，尤其是選民是否獎懲救災與防災表現。

所以模型把選舉時序當成小權重的社會訊號，而不是主因。它包含：

- 同年總統選舉。
- 同年地方選舉。
- 隔年總統選舉。
- 隔年地方選舉。
- 180 天內重大選舉。

這裡最重要的是「隔年總統選舉」。台灣總統大選常在一月，前一年颱風季可能比選舉當年颱風季更接近選舉壓力。這是模型特徵設計，不是結論。

## baseline 怎麼算

目前 `TyphoonDayBaseline` 用的是 logistic score。每個訊號加上一個可讀的 logit contribution，最後轉成 0 到 1 的機率。

目前範例權重如下：

```text
base = -2.5
storm_radius_within_4h = +1.0
mean_wind_ge_7_beaufort = +1.1
gust_ge_10_beaufort = +1.1
rainfall_24h_ge_200mm = +0.9
same_year_local = +0.25
major_election_within_180_days = +0.2
```

這些權重不是用完整歷史資料訓練出來的參數，而是把法規門檻與研究假說轉成可檢查的初始分數。它的價值是透明，不是準確率保證。

## 修正後的範例：不要把週五風雨貼到週三

早版範例有一個嚴重問題：它把高風雨情境直接貼到 2026-07-08 臺北市，導致模型輸出 88.59%。但如果颱風真正最接近的時間是週五，7/8 本身就不該沿用週五的風雨輸入。這不是模型小誤差，而是資料輸入設計錯誤。

修正後，預測改成週內 batch。每一天都要有自己的風雨情境：

| 日期 | 情境 | 機率 | 風險 |
| --- | --- | ---: | --- |
| 2026-07-08 | 颱風仍遠，未進入四小時暴風半徑情境 | 11.41% | low |
| 2026-07-09 | 外圍環流增強，但尚未達主要風力門檻 | 16.11% | low |
| 2026-07-10 | 週五最接近，風雨條件達高風險情境 | 88.59% | high |
| 2026-07-11 | 遠離後仍有殘餘雨勢與陣風 | 35.43% | low |

這個修正的重點不是宣稱週五一定放，而是把時間維度放回模型。颱風假預測不是問「這個颱風會不會讓台北放假」，而是問「哪一天、哪個縣市、在當時預報風雨下，停止上班上課機率是多少」。

輸出位置：`../generated/prediction_week_taipei.json`

## 舊範例為什麼錯

舊範例把「平均風速達七級風門檻、最大陣風達十級風門檻、24 小時雨量 240 毫米、暴風半徑四小時內可能經過」這組高風險輸入，錯放到 2026-07-08。模型因此輸出 88.59%，風險等級 high。問題不在這個機率本身，而在它被放到 7/8。若 7/8 沒有這些風雨條件，它就不應該高機率。

請注意：這仍然不是官方即時預報。它是示範「如果週五拿到這些風雨輸入，模型會怎麼拆解」。真正要預測本週，必須把最新氣象署 WPPS 預報和颱風警報資訊餵進去，並且逐日更新。

## 怎麼自己跑

在 repo 根目錄可以先跑週內 batch：

```bash
PYTHONPATH=src python3 -m typhoon_day_research.cli predict-batch \
  --input data/manual/taipei_typhoon_week_scenario.csv \
  --out reports/generated/prediction_week_taipei.json
```

如果只想跑單日，請確認該日期的風雨輸入就是那一天的情境。以週五高風險情境為例：

```bash
PYTHONPATH=src python3 -m typhoon_day_research.cli predict \
  --county 臺北市 \
  --target-date 2026-07-10 \
  --max-gust-mps 30 \
  --mean-wind-mps 16 \
  --rainfall-24h-mm 260 \
  --storm-radius-within-4h
```

也可以先解析中央氣象署 WPPS 風雨預報：

```bash
PYTHONPATH=src python3 -m typhoon_day_research.cli parse-wpps \
  --input data/raw/cwa/WPPS-Data.js \
  --out data/processed/cwa_wpps_forecast.csv
```

目前 WPPS 解析出的檔案是 `../../data/processed/cwa_wpps_forecast.csv`，包含 24 列預報區域，欄位有發布時間、平地雨量區間、山區雨量區間。

## 模型不是神諭

這個模型有幾個明確限制。

第一，它還沒有用完整歷史資料估計權重。未來應該用 logistic regression 或 hierarchical model，把歷史決策、風雨預報、實際觀測、縣市固定效果接起來。

第二，它還沒有完整處理「預報」和「實際」的落差。颱風假決策應該用放假前可得資料評估，而不是用事後天氣責備前一晚決策。

第三，它還沒有完整處理地形與交通。法規明確把交通、水電供應、地形影響納入裁量，模型如果只看風雨，很容易低估山區和離島風險。

第四，它不是官方工具。它只能當研究、科普和自我檢查用。真正決定權仍在地方政府。

## 下一版模型應該長什麼樣

下一版模型可以分三層。

第一層是即時 baseline：用 CWA 最新風雨預報、颱風警報、選舉時序、縣市歷史傾向，快速輸出透明機率。

第二層是歷史校正模型：把 2001 至今的 DGPA 決策、CWA 預報、CODiS 實際觀測合併，估計各縣市在相似風雨下的停止機率。

第三層是事件研究：針對特定颱風，比較放假前預報、放假決策、放假後實際觀測，以及輿論反應。這一層最適合寫故事，也最能說明模型為什麼不能只看單一數字。

## 結論：好的預測要讓人知道自己不知道什麼

颱風假預測最迷人的地方，是它永遠帶著不確定。颱風會轉彎，雨帶會變形，地方交通會臨時中斷，清晨資料會推翻前一晚判斷。任何聲稱百分之百準確的模型，都比天氣本身更可疑。

所以我喜歡的模型不是最跩的模型，而是最誠實的模型。它告訴你：目前機率是多少，哪些因素把機率往上推，哪些資料還缺，什麼情境下它可能錯。

颱風假的科學，不是把人類判斷消滅，而是讓判斷更透明。
